# 供应商多子配置树形建模 Design

**Goal:** 同步用户刚更新的设计稿（`docs/ui/Agent Store.dc.html`，本地未提交的修改）——本次更新把此前只应用于内置 `local` 供应商的"父级 + 可展开子配置树"模式，推广到**任意供应商**：任何已添加的供应商都能"新增子配置"（复制出一份新的独立配置，比如同一个供应商用两个不同的 API Key），子配置以缩进树形显示在父行下方。

## 背景

由于 chrome-devtools MCP 工具在本次会话中断线，这次改动是直接读取 `Agent Store.dc.html` 的模板与 `class Component` 逻辑源码（而非渲染截图）逆向出来的，由一个专门的研究 agent 逐项在源码里核实（不是仅凭 README 转述）。以下每一项都标注了置信度。

## 高置信度改动（已在模板/逻辑代码中直接核实）

1. **供应商多子配置树**：任意 `p.isProvider` 的父行都渲染一个"新增子配置"的 `+` 按钮（`p.onDuplicate`），子配置以缩进 + 连接线渲染在父行下方，每个子配置只有名称 + `×` 删除按钮（没有 local 特有的端口/圆点）。背后用一个新的 `providerParents` 映射（子 id → 根父 id）区分父子。**这与此前 `duplicateProvider` 的行为不同**：旧版复制出的条目是平级独立的 `-copy` 条目，不追踪父子关系，界面上也不会缩进嵌套。
2. **"推荐"列表恒定包含所有供应商**（即使已添加过），目的是让用户随时能给同一个供应商再建一份配置。供应商行的按钮文案是"配置"（不是"安装"），点击直接触发新增子配置逻辑；点击整行（非按钮区域）打开**只读信息页**，不是商店详情页。技能/MCP 维持原有"安装"文案与商店详情跳转。
3. **`defaultProviderConfig` 新增必填字段 `targets`**（适用客户端，claude/codex 多选）——这个字段的语义和我们已经实现的"适用客户端"图标选择完全一致（我们的实现直接调用真实的 `enable`/`disable` RPC，比设计稿把它存进配置对象里的做法更贴近真实系统状态，继续保留我们的实现方式，不追加一个重复的 `targets` 字段到 config.json 里）。
4. **"CLI 覆盖"高级设置依旧未实现**——设计稿这次甚至把原来占位用的 JSON 文本框 UI 也删掉了，只剩下一行文案，没有任何输入控件。与我们此前"不做"的裁定一致，无需变更。
5. **供应商配置表单从弹窗改成了内嵌右侧面板**——这正是此前 `2026-07-05-ui-fidelity-correction-design.md` 里记录、被裁定为"延期到独立计划"的第 4 项。这次设计稿更新进一步坐实了这个方向，但本次改动范围**不包含**这一项（弹窗→内嵌面板本身工作量大，独立处理）。
6. **图标栏确认仍是 4 个图标**（概览/供应商/技能/MCP）——与我们已实现的一致，无需改动。README 里提到的"浏览商店/更新"图标文案对应的是设计稿里一段已经不再被任何模板引用的死代码（`navDef`/`cliNav`），不是真实界面。
7. **本地代理新增端口从 `18100` 起步 +100**——设计稿字面值如此，但沿用此前已经明确决定的裁定：我们的默认端口保持 `18780`，不跟随这个具体数字对齐（`2026-07-05-priority-failover-routing.md`/`multi-port-local-relay.md` 已有明确记录）。

## 中置信度改动（源码逻辑强烈指向，但研究 agent 未能在浏览器里跑起来做最终确认）

8. **设计稿自己的"保存"校验+"配置已保存" toast 目前是死代码**——`saveCliEdit`（校验 `apiKey`/`targets` 非空、成功后 2.6 秒 toast）在整个模板里找不到任何按钮绑定它；真正在起作用的只有无校验、无提示的 `_autosave`（防抖 500ms，直接写入）。

**裁定：不跟随这处退步。** 我们已经实现的版本（`ProviderEditModal.tsx`，来自 `2026-07-05-provider-edit-form-rebuild.md` 的补救修复）同时具备防抖自动保存 **和** 校验 + "配置已保存" toast——这是设计稿自己曾经设计出来、后来在这次更新里意外/主动地把绑定点删掉了的功能，不是"设计稿现在要求我们退步成无校验"。继续保留我们已有的更完善实现，不做改动。这个决定明确记录在此，不是遗漏。

9. **代理请求日志弹窗（本地代理详情页里的入口）在设计稿源码里同样找不到绑定**——同上，不跟随退步，继续保留我们已实现的"查看代理日志"入口（Overview 卡片 + local 父详情页两处）。

## 范围决策

本次改动只做上述第 1、2 项（供应商多子配置树 + 推荐列表恒定包含供应商），因为：

- 第 3 项（`targets` 字段）本质上已经用更贴近真实系统的方式实现了，无需变更。
- 第 4、6、7 项本来就和我们已有实现一致或已有明确裁定，无需变更。
- 第 5 项（弹窗→内嵌面板）单独另开计划（`2026-07-05-inline-provider-config-panel.md`，本次不写）。
- 第 8、9 项是设计稿自己的退步，不跟随。

## "只读供应商信息页"的复用决定

设计稿新增了一个独立的"供应商信息页"内容类型（`showInfo`），用于展示父供应商的只读概览。**我们不需要新建组件**——`DetailPanel.tsx` 已经是通用的只读信息视图（对已安装项显示"✓ 已安装"而非编辑表单，对未安装项显示"安装"按钮），点击供应商父行本来就会走这条路径。本次只需要在父行是"有子配置的供应商"时，在 `DetailPanel.tsx` 里追加一行"已有 N 份配置"的提示文字（复用已有的 `installed`/父子关系数据），不需要另建页面。

## 数据模型改动

- `packages/types/src/engine.ts`：`InstalledItem` 新增 `parentSlug?: string`（省略表示是根条目）。
- `apps/client-core/src/engine.ts`：`duplicateProvider(slug)` 计算根父 id（若 `slug` 自己有 `parentSlug` 则用它的根，否则用 `slug` 自己），写入新条目的 `parentSlug`。
- `apps/client-core/src/config/provider.ts`：`duplicateProviderConnection` 复制配置时**清空 `apiKey`**（不再整份复制），保留其余字段（`baseUrl`/`homepage`/`level` 等作为起点更方便）。

## 不做的事（本次范围外）

- 供应商配置表单从弹窗改内嵌面板——另开计划。
- 设计稿死代码里暗示的校验退步/代理日志弹窗退步——明确不跟随。
- "只读供应商信息页"单独建组件——复用 `DetailPanel.tsx`。
